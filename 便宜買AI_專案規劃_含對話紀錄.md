## 產品定位

### 核心定位
**便宜買 AI = 消費者採購助手 + 節氣飲食指南**

幫助家庭消費者根據 24 節氣、產地天氣、市場行情，做出更聰明的蔬果採購決策。

### 目標用戶
- 家庭主婦/主夫
- 注重飲食健康的消費者
- 對節氣飲食有興趣的用戶

### 商業模式（V1）
- 免費使用 + 廣告收益
- 未來可升級為「高級預測」訂閱服務

## 回報菜價與區域預測機制

### 核心邏輯：「回報 → 受惠」正反饋迴圈

```
用戶 A：「我在台中買絲瓜 $38 / 菜市場」
   ↓ (一鍵回報，10 秒完成)
系統：聚合同地區其他用戶數據
   ↓
AI 分析：「台中絲瓜通常比全國平均便宜 5-8%」
   ↓
推薦給用戶 A：
  • 下週預測：$35-42/kg
  • 🟢 現在買划算（預計下週漲）
  • 同區還有 23 人回報相同數據
   ↓
網絡效應：數據越多 → 預測越準 → 用戶越信任 → 更多人回報
```

### 預測演算法（V1 簡易版）

```python
def predict_regional_price(item, region, date):
    # 1. 基準：MOA 全國平均
    moa_avg = get_moa_price(item, date)

    # 2. 地區修正因子（用戶眾包）
    regional_reports = get_user_reports(item, region, days=30)
    if len(regional_reports) > 5:
        regional_factor = mean(regional_reports) / moa_avg
    else:
        regional_factor = 1.0  # 數據不足，用全國平均

    # 3. 趨勢（7 天移動平均）
    trend = simple_moving_average(get_historical(item, region, days=60), window=7)

    # 4. 節氣因子
    seasonal_mult = get_seasonal_multiplier(item, date)

    # 5. 融合預測
    predicted = moa_avg * regional_factor * trend * seasonal_mult

    return {
        "prediction": predicted,
        "confidence": len(regional_reports),
        "range": [predicted * 0.9, predicted * 1.1],
        "regional_offset": (regional_factor - 1) * 100
    }
```

### 數據質量控制

| 層級 | 機制 | 說明 |
|------|------|------|
| 提交前 | 邊界檢查 | 價格範圍驗證（e.g., 絲瓜 $5-100） |
| 提交時 | 立即反饋 | 「這個價格比全國高 50%，確定嗎？」 |
| 聚合後 | 異常偵測 | 去除 >3σ 離群值 |
| 長期 | 信譽積分 | 活躍回報者數據權重更高 |

## 雙版本規劃

### 策略：分階段推進（先消費者，後農民）

```
Phase 1（W1-W6）：消費者版「便宜買 AI」
├─ 上線消費者應用
├─ 完成共用基礎設施（MOA 資料、節氣資料庫、預測演算法）
├─ 累積用戶數據與回報
└─ 獲得市場反饋

Phase 2（W7-W12，3 個月後）：農民版「優値情報站」
├─ 基於 Phase 1 用戶數據優化預測
```

### 兩版本數據協同

```
消費者回報：「台中絲瓜 $38」
   ↓ 聚合
消費者版：「下週可能 $40，現在買划算」
農民版：「台中地區需求穩定，建議種植」

農民在互助網：「種太多絲瓜，急銷」
   ↓ 系統偵測供應充足
消費者版：「今週絲瓜特別便宜！」
   ↓ 消費者湧入購買 → 幫助農民解決滯銷
```

## 數據架構

### 資料表

```sql
-- 批發價格（MOA 官方數據）
CREATE TABLE wholesale_prices (
    id UUID PRIMARY KEY,
    product_name VARCHAR(100),
    origin VARCHAR(100),
    market VARCHAR(100),
    date DATE,
    price_low NUMERIC,
    price_high NUMERIC,
    price_avg NUMERIC,
    source_id VARCHAR(50),
    created_at TIMESTAMP
);

-- 用戶價格回報（眾包數據）
CREATE TABLE user_price_reports (
    id UUID PRIMARY KEY,
    product_name VARCHAR(100),
    price_reported NUMERIC,
    location_county VARCHAR(50),
    market_type VARCHAR(50),
    purchase_date DATE,
    quality VARCHAR(20) DEFAULT 'pending',
    created_at TIMESTAMP
);

-- 區域價格統計快取
CREATE TABLE regional_price_stats (
    id UUID PRIMARY KEY,
    product_name VARCHAR(100),
    region VARCHAR(50),
    date DATE,
    avg_price NUMERIC,
    report_count INT,
    std_dev NUMERIC,
    updated_at TIMESTAMP
);

-- 價格預測結果快取
CREATE TABLE price_predictions (
    id UUID PRIMARY KEY,
    product_name VARCHAR(100),
    region VARCHAR(50),
    predicted_date DATE,
    predicted_price NUMERIC,
    confidence_score FLOAT,
    trend_direction VARCHAR(10),
    created_at TIMESTAMP
);
```

### 最容易卡關的地方

**W1 — MOA 資料格式**
政府開放資料常有 Big5/UTF-8 編碼問題，建議先手動下載一份確認格式再寫 ETL。

**W3 — 回報數據不足**
早期用戶少時地區統計數據不足，預測信心低。建議 MVP 先顯示「全國平均」作為 fallback，等數據累積後再顯示地區預測。

**W5 — 預測演算法**
不需要複雜的 ML。MVP 用「7 天移動平均 × 節氣因子」就夠，先上線驗證，V2 再升級模型。

### 討論一：天氣收集是否需要新增到介面？

**問：** 天氣收集主要是為了預測後續菜價，有需要新增到介面嗎？

**結論：**

依 MVP 時程（6 週）與資源，採用**方案 A：暫不收集天氣數據**。

- 時間限制：天氣數據預測是複雜的機器學習工作，超出 V1 範圍
- 業務優先級：核心是「菜價查詢 + 互助網」，不是天氣預報
- 文件已明標：進階分析為「V3+」範疇

**替代方案：方案 B（收集但不展示）**
若未來要預留，可建立 `weather_data` 表每日同步 CWA 資料，不在 UI 呈現，待 V3 實裝預測功能時使用。

**決策：V1 不收集天氣資料，V3 規劃天氣 × 菜價關聯分析。**

---

### 討論二：節氣指南有存在的必要嗎？

**問：** 網站內有個叫節氣指南，有存在的必要嗎？

**背景釐清：**

發現專案存在兩個不同定位的產品：

| | 便宜買 AI | 優値生鮮情報站 |
|------|---------|------------|
| 類型 | 消費者應用（B2C） | 農民決策支援平台（B2B） |
| 用戶 | 家庭購物者 | 小農、農民、合作社 |
| 節氣指南 | ✅ 有存在必要 | ❌ 不應該有 |

**結論：**

- **便宜買 AI（消費者版）**：節氣指南是核心亮點，應保留並強化
  - 節氣 × 推薦食材 × 即時菜價 = 差異化功能
  - SEO 高搜尋量關鍵詞（立夏買什麼菜、節氣飲食）
- **優値情報站（農民版）**：不需要節氣指南，改為「農產新知」

---

### 討論三：確認主要開發方向

**問：** 重新整理了資料，網站內是節氣指南（確認是「便宜買 AI」消費者應用）

**問：** 消費者為主

**結論：**

確認以「便宜買 AI」消費者應用為主要開發目標。產品定位：
- 核心用戶：家庭購物者、消費者
- 核心價值：聰明買菜 + 節氣飲食知識
- 節氣指南：保留並作為核心差異化功能

---

### 討論四：回報菜價功能重新定位

**問：** 回報菜價可以為消費者打造結合市場價格和自己區域的菜價做預測，這樣如何？

**分析：**

這把「回報菜價」從單純的數據貢獻升級成**個性化預測引擎**。

**商業價值對比：**

| 維度 | 沒有回報功能 | 有回報功能 |
|------|----------|----------|
| 用戶價值 | ⭐⭐⭐ 被動搜尋 | ⭐⭐⭐⭐⭐ 主動參與 + 個性化預測 |
| 黏著度 | 低 | 高（每週貢獻 + 看預測） |
| 數據質量 | MOA 全國平均 | 眾包 + AI = 「你的台中菜價」 |
| 差異化 | 無 | 獨有亮點 |
| 網絡效應 | 無 | ✅ 人越多預測越準 |

**結論：保留回報菜價功能，以「回報 → 預測」正反饋設計取代單純義務型回報。**

---

### 討論五：加入農民版的影響評估

**問：** 加入農民板會有什麼區別或新增？

**時間成本分析：**

| 方案 | 開發時間 | 建議 |
|------|---------|------|
| 只做消費者版 | 6 週 | ✅ |
| 同步做雙版本 | 14 週 | ❌ 風險高 |
| 分階段做 | 12 週（分別上線） | ⭐ 推薦 |

**農民版新增工作（Phase 2）：**
- 認證系統（OAuth）
- 訂閱管理（金流）
- 互助網平台（發佈/審核）
- 農民儀表板（售價分析）
- LINE Notify 整合
- 推播排程系統

**兩版本協同價值：**
消費者回報數據 → 農民了解地區需求  
農民互助網（滯銷）→ 消費者得到「今週特別便宜」提示

**結論：採用分階段策略，先做消費者版驗證市場，3 個月後再推農民版。**
